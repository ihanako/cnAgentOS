# 架构与产品决策记录

## 2026-05-27：旧材料只用于提取功能需求

**决定**：旧的 Markdown/TXT 文档仅用于确认系统想解决的问题和候选方向，不继承其中的 API、数据表、完成状态、技术细节或敏感配置。

**原因**：旧材料描述的是示例原型，且不同文档之间存在完成度和实现口径冲突。

**影响**：正式需求与开发契约统一维护在 `docs/`。

## 2026-05-27：示例代码允许完全重写

**决定**：旧应用代码、旧数据库与旧文档在需求提取完成后废弃并移出正式工程；正式实现按新契约重新建设。

**原因**：目标是构建符合正式需求、安全边界和可持续开发规范的新系统，而不是维护示例兼容性。

## 2026-05-27：首版采用 Python 模块化 MVC 单体

**决定**：正式版本使用 Python 后端，按 MVC 与业务模块组织为单体应用；首版不引入微服务。

**原因**：当前核心风险在于完成数据采集到智能问数的可靠闭环。单体能降低部署、分布式一致性和跨服务鉴权成本，并保持模块边界清晰。

**重新评估条件**：出现模块独立扩缩容、采集影响主站稳定性、独立团队交付边界或强隔离需求。

## 2026-05-27：Python 工程统一使用 uv 管理

**决定**：项目统一使用 `uv` 管理 Python 版本、项目依赖、本地虚拟环境和依赖锁定；`pyproject.toml` 描述依赖范围，`uv.lock` 固定团队与交付使用的解析结果，两者均纳入版本控制。

**原因**：三人并行开发需要一致且可复现的依赖环境，避免成员分别使用不同安装方式产生依赖漂移、验证结果不一致或交付环境不可复现。

**影响**：成员通过 `uv sync` 同步环境，通过 `uv add` / `uv remove` 调整依赖，通过 `uv run ...` 执行项目命令，并使用 `uv lock --check` 检查锁文件一致性。除后续部署集成明确要求外，不另行维护与 `pyproject.toml` / `uv.lock` 并行的依赖真源。

## 2026-05-27：首版范围锁定核心闭环

**决定**：首版包含认证/RBAC 与动态功能导航、模型引擎、智能瞭望、数据仓库、智能问数和必要安全审计。

**不包含**：即时通讯、数字员工、舆情大屏、语音/手势、复杂自动化和多数据库切换。

## 2026-05-27：新数据库与 API 文档为正式契约

**决定**：`docs/database/` 与 `docs/api/` 描述的是目标重写系统需要实现的接口和数据规则，后续代码应围绕这些契约开发。

**变更方式**：业务决定发生变化时，必须同步更新契约与本决策记录，而不是仅在代码中产生隐性差异。

## 2026-05-27：变更必须经过分支和 Code Review 流程

**决定**：正式变更采用 Conventional Commits；不得直接在 `main`、`master` 或其他受保护主分支提交；变更应在工作分支完成并由项目负责人 Code Review 后合并。

**原因**：项目将持续由人类与 AI 协作开发，需要通过可审查提交和人工合并门禁防止未经确认的设计或实现进入主线。

**影响**：AI 默认只能准备工作分支变更和评审材料，不得自行合并主分支；完整规则维护在 `docs/workflow/git.md`。

## 2026-05-27：以 MVP 文档和 Phase 计划组织三人协作

**决定**：新增 `docs/product/mvp.md` 作为首版验收边界，新增 `docs/planning/development-plan.md` 将实施拆分为可并行的 Phase 和三条责任流。

**原因**：正式需求描述能力范围，但三名开发者还需要明确交付判据、依赖顺序、并行边界与评审节点，才能减少共享文件冲突和重复设计。

**影响**：每项开发任务应关联一个 MVP 旅程和当前 Phase 工作包；Phase 完成以验收和 Code Review 合并为准。

## 2026-05-27：Phase 1 后端采用 FastAPI 异步 PostgreSQL 技术栈

**决定**：应用底座采用 FastAPI、Pydantic v2、SQLAlchemy 2 `AsyncSession`、Alembic、PostgreSQL 与 `asyncpg`；本地开发通过 Docker Compose 提供 PostgreSQL；测试使用 `pytest` 与 `httpx`。

**原因**：正式接口以 JSON API 和后续 SSE 为核心，权限依赖需供各模块复用；PostgreSQL 能直接承载关系约束并为后续内容检索提供稳定演进路径。

**影响**：B 后续在同一迁移链新增模型、采集和问数表；C 通过 FastAPI 接口契约联调，不另建身份逻辑。

## 2026-05-27：认证采用服务端会话、同步 CSRF 与一次性管理员引导

**决定**：密码采用 Argon2id 且长度为 12 至 128 个字符；登录使用数据库持久化的不透明会话令牌哈希，默认有效期 8 小时；变更请求使用 `X-CSRF-Token`，登录与 `GET /api/v1/auth/me` 返回当前令牌；首个系统管理员通过一次性 CLI 创建。

**原因**：该方案支持服务端即时撤销会话和权限变更生效，避免在仓库、迁移或 API 中暴露初始密码与会话秘密，并允许后台页面刷新后安全继续操作。

**影响**：普通用户仅由管理员创建，首版不提供公开注册；生产部署必须配置专用 CSRF secret 并启用安全 Cookie。

## 2026-05-28：采用单仓前后端分离开发边界

**决定**：仓库保持一个 Git 工作区，后端工程移动到 `backend/`，前端工程位于 `frontend/`，共享产品、API、数据库、计划和上下文文档继续保留在根目录 `docs/`。MVP 阶段采用前后端分离开发，但不拆成两个仓库。

**原因**：首版仍需要后端 API、服务端会话、CSRF、SSE 和页面联调保持同一评审边界；单仓可降低接口同步和验收成本，同时让前后端源代码边界清晰。

**影响**：Python 命令在 `backend/` 下执行，前端命令通过 `pnpm --dir frontend ...` 执行；根目录 `compose.yaml` 继续提供本地 PostgreSQL。后端进程只提供 API、健康检查和 OpenAPI 文档，不托管前端页面或 `frontend/dist` 静态资源；开发期由 Vite 代理 `/api` 到 FastAPI。只有出现强隔离合规、独立团队边界或跨仓发布管理需要时，才重新评估双仓。

## 2026-05-28：前端采用 Vite Vue 脚手架与 Element Plus

**决定**：前端使用 Vite `vue-ts` 脚手架生成，技术栈为 Vue 3、TypeScript、Vue Router、Pinia、Element Plus、Vitest、ESLint 和 Prettier；依赖和锁文件由 `pnpm` 管理。

**原因**：MVP 管理端以表格、表单、弹窗、权限选择和流式状态展示为主，Element Plus 能覆盖通用后台交互；智能问数和引用展示保留自定义业务组件空间。

**影响**：开发时 Vite 代理 `/api` 到 FastAPI；后端不托管前端构建产物。

## 2026-05-28：模型调用采用 OpenAI Python SDK

**决定**：后端模型引擎使用官方 `openai` Python SDK 的 `AsyncOpenAI` 调用 OpenAI-compatible Chat Completions；前端继续只消费后端 `/api/v1` 和 SSE，不引入 OpenAI 前端 SDK 或 Vercel AI SDK。

**原因**：模型凭据、RBAC、审计、调用日志、SSE 编排和后续引用持久化均属于 FastAPI 后端职责；使用 Python SDK 能减少手写 HTTP/SSE 解析，同时保持现有 OpenAI-compatible 契约。

**影响**：`openai` 是后端生产依赖；`httpx` 仅保留为后端测试开发依赖。后续普通问数和流式问数应复用模型 provider 适配器，不在 Controller 中直接拼接上游 `/chat/completions` 请求。

## 2026-05-28：Phase 2 采集安全采用保守 SSRF 策略

**决定**：首版采集来源只允许 HTTPS；`allowed_hosts` 采用精确主机名匹配，不支持通配符、协议、端口或路径。数据源保存、任务执行和重定向每一跳都必须执行服务端安全校验，拒绝 URL userinfo、本地域名、回环/私网/链路本地/保留/组播/云元数据地址，以及 DNS 解析中出现任一非公网地址的目标。

**原因**：Phase 2 的采集能力会主动访问外部地址，必须先把 SSRF、防内网探测和敏感失败信息泄露风险收紧，再交给数据源、规则和任务执行模块复用。

**影响**：Phase 2 B 的数据源保存、启用、任务创建、任务执行和重定向处理必须调用采集安全组件；若后续确需开发环境 HTTP 或内网例外，需要先更新安全契约和本决策记录。

## 2026-05-29：问数会话隔离采用查询级所有权过滤

**决定**：Phase 3 智能问数的会话、回答和引用访问必须在数据库查询条件中绑定当前用户，不允许先按 ID 查出记录再在 Python 中判断归属。访问他人会话、他人回答或不存在记录均对外返回 `404 NOT_FOUND`。

**原因**：问数会话会沉淀用户问题、回答和引用依据，属于普通用户个人数据范围；统一不可见错误可以避免通过 ID 探测其他用户会话是否存在。

**影响**：Phase 3 B 的问数会话 API、消息 API、引用 API 和 SSE 提问入口必须复用 `qa_security` helper；如后续新增管理员查看全局问数记录的能力，必须另行设计专用后台权限和审计规则。
